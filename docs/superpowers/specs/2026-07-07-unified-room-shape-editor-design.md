# Єдиний редактор форми кімнати — дизайн

Дата: 2026-07-07
Статус: затверджено користувачем, готово до `writing-plans`

## Контекст і мотивація

Це підпроєкт (A) з трьох, на які розбито велике прохання користувача
("проєкт мрії" — переробити редактор геометрії кімнати повністю):

- (A) **Єдиний редактор форми кімнати** — цей документ.
- (B) Розширені прорізи (несучі стіни, вентиляція, технічні отвори, арки) — окремий спек пізніше.
- (C) 3D-візуалізація кімнати (статичне ізометричне прев'ю + опціонально інтерактивна 3D-модель,
  залежно від можливостей пристрою) — окремий спек пізніше.

(B) і (C) залежать від даних, які видає (A), тому починаємо з (A).

### Що не так із поточним редактором (підтверджено на пристрої 2026-07-07)

Поточний екран геометрії кімнати має три окремі вкладки (Прямокутник /
Довільний контур / Малювати), кожна зі своїм станом:

1. Перемикання Прямокутник → Довільний контур губило реальні розміри
   (4000×3000 ставало 3000×3000 — квадрат), бо `edges` бралось зі
   статичного дефолту, а не з поточних `widthMm`/`lengthMm`.
2. Вкладка "Малювати" завжди починалась з порожньої сітки, навіть якщо
   кімната вже мала форму — існуючий контур ніде не підвантажувався.
3. Взаємозв'язку між кутом і сусідніми стінами не було: кожне поле
   (довжина, кут) — незалежний текстовий інпут. Єдиний натяк на
   взаємозв'язок — пасивне попередження про "нев'язку контуру" вже
   ПІСЛЯ введення, без live-перерахунку.

Ці баги — симптом глибшої архітектурної проблеми: геометрія зберігається
як послідовність "довжина+кут" (`Edge`), яку `GeometryEngine.buildPolygon()`
"проходить" від умовної точки (0,0), накопичуючи похибку, яку потім треба
чи то виправляти автокорекцією, чи блокувати збереження. Це і є джерело
"нев'язки контуру" як окремого класу проблем.

## Рішення: одна модель, один екран

### Зміна моделі даних

`RoomGeometry` зводиться до одного типу:

```kotlin
sealed class RoomGeometry {
    data class Polygon(val vertices: List<Vertex>) : RoomGeometry()
}
```

`Rectangle` як окремий тип видаляється. Прямокутник — це просто початковий
стан вершин нової кімнати (4 точки, 4000×3000мм), а не окремий режим
зберігання.

**Замкнутість контуру стає структурною гарантією**: список вершин по колу
не може бути "розірваним". Тому видаляються:
- `GeometryEngine.buildPolygon(edges)`
- `ClosureStatus`, `PolygonResult.closureErrorM/status`
- `distributeClosureError()`
- Усі попередження типу "нев'язка контуру N см"

**Міграція даних:** без міграції. Застосунок ще не в Google Play (є активні
блокери релізу — package name, storage.rules), реальних користувачів з
уже збереженими кімнатами немає. Формат просто змінюється.

### Похідні операції над вершинами (нове в `GeometryEngine`)

Всі приймають поточний `vertices: List<Vertex>` і повертають новий список
(чиста функція, без побічних ефектів):

```kotlin
fun moveVertex(vertices: List<Vertex>, index: Int, newPosition: Vertex): List<Vertex>

fun insertVertexOnEdge(vertices: List<Vertex>, edgeIndex: Int, atPoint: Vertex): List<Vertex>

fun removeVertex(vertices: List<Vertex>, index: Int): List<Vertex>
// вимагає vertices.size > 3, інакше кидає/повертає null — рішення на рівні ViewModel

fun setEdgeLength(vertices: List<Vertex>, edgeIndex: Int, newLengthMm: Int): List<Vertex>
// рухає vertices[edgeIndex + 1] вздовж поточного напрямку стіни edgeIndex,
// vertices[edgeIndex] лишається нерухомим. Довжина наступної стіни (edgeIndex+1)
// змінюється як похідний наслідок (обидва її кінці визначені: нова позиція
// vertices[edgeIndex+1] і нерухома vertices[edgeIndex+2]).

fun setInteriorAngle(vertices: List<Vertex>, vertexIndex: Int, newAngleDeg: Double): List<Vertex>
// vertices[vertexIndex - 1] лишається нерухомим, довжина стіни
// (vertexIndex-1 -> vertexIndex) лишається незмінною. vertices[vertexIndex]
// обертається навколо vertices[vertexIndex - 1] на різницю (new - current)
// кута, зберігаючи напрямок обходу (щоб полігон не "вивернуло").
// vertices[vertexIndex + 1] лишається нерухомим — довжина стіни
// (vertexIndex -> vertexIndex+1) змінюється як похідний наслідок.
```

Лишаються без змін (вже працюють з вершинами, формат зберігання їх не
стосується): `polygonAreaM2`, `perimeterM`, `hasSelfIntersection`,
`isConvexCorner`, `countConcaveCorners`, `wallAreaGross/Net`,
`transformVertices`, `closestPointOnSegment`, `projectionRatio`,
`beaconsCountForWall`.

`verticesToEdges()` лишається, але міняє роль: тепер це чиста
DISPLAY-функція для таблиці чисел (довжина+кут кожної стіни, похідні від
vertices), а не формат зберігання.

### Екран: канвас + таблиця чисел, синхронні в обидва боки

Один екран замість трьох вкладок:

- **Канвас** (зверху): вершини — кружечки, стіни — лінії з підписом
  довжини. Нова кімната відкривається з дефолтним прямокутником
  (4000×3000, 4 вершини) — не порожньою сіткою.
  - **Перетягнути вершину** → вільний рух, решта кімнати нерухома.
    Це і є "зміна кута" в дії — сусідні стіни самі перерахують довжину.
  - **Короткий тап по стіні** (не по вершині) — лишається як зараз:
    відкриває діалог додавання проріму (вікно/двері/тощо). Не змінюється,
    щоб не ламати вже робочий і звичний жест.
  - **Довге натискання по стіні** (не по вершині) → вставляє нову вершину
    в точці натискання (`insertVertexOnEdge`), ділить стіну на дві.
  - **Довге натискання на вершину** → пропонує видалити (`removeVertex`).
    Мінімум 3 вершини завжди. Якщо на одній із двох суміжних стін є
    проріз — видалення **блокується** з повідомленням "Спершу приберіть
    проріз(и) на суміжній стіні".
  - Однакова логіка для обох дій (вставка/видалення вершини) — не два
    окремі перемикачі режимів, а розрізнення за тривалістю натискання.
  - Живий self-intersection check (`hasSelfIntersection`) — червоне
    попередження, перетягування не забороняється фізично, але збереження
    блокується, поки перетин не усунуто (як і зараз).
  - Полотно саме підганяє масштаб під розмір кімнати (fit-to-view: скейл
    = min(доступна ширина/розмах X, доступна висота/розмах Y) з відступами).
    Масштаб заморожується на час одного жесту перетягування (обчислюється
    в `onDragStart`, використовується без перерахунку до `onDragEnd`), щоб
    точка не "стрибала" під пальцем через зміну bounding box на льоту.
- **Таблиця чисел** (знизу): для кожної стіни — довжина (мм) і внутрішній
  кут (°), обидва редаговані текстові поля. Редагування довжини викликає
  `setEdgeLength`, редагування кута — `setInteriorAngle`. Кнопка
  "+Стіна" (додає вершину на середині останньої стіни — зручний шлях без
  точного тапу по канвасу для тих, хто не хоче цілитись пальцем).

### Прорізи при вставці/видаленні вершини

При `insertVertexOnEdge(edgeIndex, atPoint)`:
- Прорізи на стінах з індексом **> edgeIndex** зсувають `wallEdgeIndex`
  на +1 (з'явилась нова стіна в списку).
- Прорізи, що були на стіні `edgeIndex`: якщо `offsetMm` проріму менший за
  відстань від `vertices[edgeIndex]` до `atPoint` — лишається на стіні
  `edgeIndex` (offset не змінюється). Якщо більший — переїжджає на нову
  стіну `edgeIndex + 1` з `offsetMm` = старий offset мінус ця відстань.

При `removeVertex(index)`: якщо на стіні `(index-1, index)` або
`(index, index+1)` є хоч один проріз — операція блокується (діалог з
поясненням), вершина не видаляється.

### Зникає з UI/логіки

- Вкладки "Прямокутник" / "Довільний контур" / "Малювати" (порожній
  аркуш) — весь функціонал зливається в один екран вище.
- "Малювати за фото плану" (`FloorplanDrawFlow`) — **лишається**, як
  альтернативний спосіб задати початкову форму (кнопка "Імпортувати фото
  плану" поруч із дефолтним прямокутником при створенні кімнати). Після
  обведення контуру по фото результат потрапляє в той самий єдиний
  редактор (канвас + числа), а не в окрему підсистему.
- `ClosureStatus`, "нев'язка контуру N см" — попередження видаляються,
  бо втрачають сенс.

### Вплив на інші файли (для плану реалізації)

- `RoomGeometry` (модель) — `Rectangle` видаляється, `Polygon` міняє поле
  `edges: List<Edge>` на `vertices: List<Vertex>`.
- `GeometryViewModel` — переписується під єдиний стан `vertices`, без
  `isRectangle`/`widthMm`/`lengthMm`/`edges: List<EdgeInput>` дуалізму.
- `GeometryScreen.kt` — прибрати перемикач вкладок Прямокутник/Довільний
  контур/Малювати, лишити один екран.
- `FreehandDrawCanvas.kt` — замінюється новим `RoomShapeCanvas` (drag +
  tap-to-insert + long-press-to-remove замість "намалювати з нуля").
- `LayersViewModel.computeArea()`, `SummaryViewModel` — прибрати
  `when (room.geometry) { Rectangle -> ...; Polygon -> ... }`, лишити
  тільки полігон-гілку.
- `RoomPlanCanvas.kt` — читає вершини напряму, без `buildPolygon`.
- `RoomRepository`/Room entity/Firestore-серіалізація `RoomGeometry` —
  оновити під нову форму (без міграції старих записів).
- `DemolitionCalculator`/інші споживачі `roomGeometryVertices()` — без
  змін у сигнатурі, лише прибрати `Rectangle`-гілку всередині.
- `GeometryEngineTest` — переписати: прибрати тести `buildPolygon`/
  `distributeClosureError`, додати тести на нові vertex-операції.

## Поза межами цього спека

- Несучі стіни, розширені типи прорізів (арки, технічні отвори) — підпроєкт (B).
- 3D-візуалізація — підпроєкт (C).
- Pinch-zoom/pan на канвасі для дуже великих кімнат — можливе майбутнє
  покращення, не блокує (A).
