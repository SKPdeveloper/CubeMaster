# CubeMaster — мобільний калькулятор будівельника

Технічна специфікація для реалізації через Claude Code. Документ — джерело істини: при розбіжності коду і спеки переписуємо код, а не патчимо його поверх.

Назва **«CubeMaster»** (UA-локалізація в сторах — «КубМайстер») — пряма відсилка до кубатури обʼємів матеріалів, тобто до того, що застосунок реально рахує, без фольклорного пафосу. Латиниця в основній назві — свідомий вибір для сторів (пошук, впізнаваність, відсутність проблем з транслітерацією в App Store/Google Play), українська локалізована версія — лише для лістингу і в самому додатку як альтернативне відображення назви.

## 0. Платформа і стек

- **Мова**: Kotlin 2.x, 100% нативний Android (Jetpack Compose, без крос-платформних компромісів на кшталт Flutter/React Native — важливо для продуктивності на слабких пристроях і для стабільної роботи офлайн-обчислень на об'єкті).
- **Мінімальна версія**: `minSdk 26` (Android 8.0) — покриває практично весь ринок будівельної техніки в Україні станом на 2026, `targetSdk` — актуальний на момент білда.
- **UI**: Jetpack Compose + Material 3 як база компонентів, поверх якої накладається власна дизайн-система (§9) — Material 3 не використовується "з коробки" візуально, лише як інфраструктура (theming, accessibility, компонентна модель).
- **Архітектура**: MVVM + Clean Architecture (`domain` / `data` / `presentation` модулі), Hilt для DI, Kotlin Coroutines + Flow для асинхронності.
- **Локальне сховище**: Room (SQLite) — джерело істини на пристрої, працює повністю офлайн. Обчислювальне ядро (`cubemaster-core`) — чистий Kotlin-модуль без Android-залежностей (можна підключити як Gradle-модуль і в майбутній desktop/CLI-версії), покритий unit-тестами.
- **Хмара — Firebase**:
  - **Firebase Authentication** (Email/Password + Google Sign-In; телефонна автентифікація — опційно, з урахуванням нестабільного мобільного звʼязку в прифронтових регіонах, тому email/password — основний, а не допоміжний метод).
  - **Cloud Firestore** — синхронізація проєктів між пристроями (прораб на об'єкті — телефон, прораб у офісі — планшет), офлайн-персистентність Firestore (`setPersistenceEnabled`) поверх Room як другий рівень кешу для мультипристроєвого сценарію.
  - **Cloud Storage for Firebase** — фотофіксація приміщень, додані до проєкту (перед/після демонтажу, фото дефектів стін для звіту).
  - **Cloud Functions (Node.js/TypeScript)** — серверна логіка, яку не можна довірити клієнту: агрегація цін з зовнішніх джерел (§8), генерація PDF кошторису (важка операція, виноситься з мобільного пристрою), scheduled-джоба оновлення прайсів.
  - **Firebase Cloud Messaging** — сповіщення про оновлення цін, готовність кошторису.
  - **Firebase Remote Config** — фіче-флаги і редаговані без релізу нормативні константи (див. §2 — коефіцієнти витрати матеріалів як remote-config values з локальним fallback, щоб оновлення норм (наприклад, після перегляду ДБН) не вимагало оновлення застосунку).
  - **Crashlytics + Performance Monitoring**.
- **Синхронізація офлайн→онлайн**: `WorkManager` — черга відкладеної синхронізації Room→Firestore з конфлікт-резолюцією "останній запис виграє за `updatedAt`" на рівні документа (проєкт/кімната), з попередженням користувачу при виявленому конфлікті між пристроями, а не мовчазним перезаписом.
- **Формат чисел**: метрична система; десятковий роздільник — кома в UI (`java.util.Locale("uk", "UA")`), крапка у сховищі/JSON.
- **Мова інтерфейсу і код**: інтерфейс — **українською** повністю (не лише рядки, а й тексти помилок, підказки, назви одиниць). Коментарі, докстрінги (KDoc), повідомлення логів у коді — **українською**. Ідентифікатори (класи, змінні, поля Firestore) — англійською, стандартна практика.

## 1. Нормативно-правова база (науково-технічне обґрунтування)

Усі константи витрати матеріалів і граничні товщини шарів у §5 не є довільними — кожна прив'язана до чинного нормативного документа України. Це принципова відмінність від типових "калькуляторів з інтернету", які беруть середні цифри з маркетингових матеріалів виробників без опори на державні норми. Посилання зберігаються прямо в моделі даних (`ConsumptionNorm.normativeReference`) і показуються користувачу за запитом ("чому саме так?") — прозорість розрахунку критична для довіри інженера до інструменту.

Ключові документи, на які спирається розрахунковий рушій:

- **ДБН В.2.6-22:2001** «Конструкції будинків і споруд. Улаштування покриттів із застосуванням сухих будівельних сумішей» — основний документ для стяжок, штукатурок, наливних підлог, облицювання. Регламентує: класифікацію сумішей за групами (штукатурні Ш2/Ш4, для підлог П1–П3, гідроізоляційні Г1/Г2), допустимі відхилення товщини елемента підлоги від проєктної (не більше 10% в окремих місцях, п. 5.20), терміни витримки самовирівнювального шару перед укладанням покриття (72 год для плитки/лінолеуму на водних клеях, 7 діб на органічних розчинниках, 28 діб під паркет), мінімальну міцність на стиск для наливних підлог під епоксидні/поліуретанові покриття (не менше 40 МПа).
- **ДСТУ Б В.2.7-126:2011** «Суміші будівельні сухі модифіковані. Загальні технічні умови» — технічні вимоги до сухих сумішей для підлог, розчинових сумішей, класифікація за крупністю заповнювача (до 1,25 мм для фінішних шарів, до 4,0 мм для базових).
- **СНиП 3.04.01-87** «Изоляционные и отделочные покрытия» — досі чинний в частині, не перекритій новими ДБН, для допустимих відхилень обштукатурених поверхонь (ДБН В.2.6-22 п. 6.6 прямо на нього посилається).
- **ДБН В.2.6-33:2018** «Конструкції зовнішніх стін із фасадною теплоізоляцією» — для розрахунку утеплення (§5.10).
- **ДБН В.1.2-14:2018** «Загальні принципи забезпечення надійності та конструктивної безпеки будівель, споруд, будівельних конструкцій та основ» — базове обґрунтування для попереджень модуля демонтажу про несучі конструкції (§6).
- **ДСТУ-Н Б В.2.6-34:2008** (та профільні настанови) — для допусків геометрії приміщень і оцінки кривизни стін під маяки.
- Технічна документація виробників (ТУ У, технічні картки Ceresit/Knauf/Anserglob/Siltek) — використовується як **уточнююче**, а не первинне джерело для конкретних торгових позицій (кожен виробник у межах норми ДБН має власну рецептуру і трохи інші цифри витрати — тому в каталозі матеріалів норма ДБН є нижньою/верхньою межею "коридору правдоподібності", а конкретна витрата з ТУ виробника — фактичне значення за замовчуванням для SKU).

Мінімальні товщини шарів у §5 — це не рекомендація "щоб добре трималось", а прив'язка до конкретних пунктів норм (наприклад, мінімум 30 мм для цементно-піщаної стяжки в житлових приміщеннях і обов'язкове армування при товщині понад 50 мм — стандартна вимога, що фігурує в чинних будівельних нормах і технічних картках, узгоджених з ДБН В.2.6-22). Це відображено в моделі `ConsumptionNorm` полем `minThicknessJustification: String` — конкретне формулювання норми, а не просто число.

## 3. Геометричний рушій (`cubemaster-core`, Kotlin, без Android-залежностей)

### 3.1 Побудова полігону з довжин сторін і кутів

```kotlin
data class Vertex(val x: Double, val y: Double)
data class PolygonResult(val vertices: List<Vertex>, val closureErrorM: Double)

fun buildPolygon(edges: List<Edge>): PolygonResult {
    // edges[i].lengthMm — довжина ребра i (від вершини i до вершини i+1)
    // edges[i].interiorAngleDeg — внутрішній кут ПРИ вершині i
    // Перше ребро йде вздовж осі X (heading = 0), обхід — за годинниковою стрілкою
    val n = edges.size
    val vertices = mutableListOf(Vertex(0.0, 0.0))
    var heading = 0.0
    var x = 0.0
    var y = 0.0
    for (i in 0 until n) {
        val lengthM = edges[i].lengthMm / 1000.0
        x += lengthM * cos(heading)
        y += lengthM * sin(heading)
        vertices.add(Vertex(x, y))
        if (i + 1 < n) {
            val exterior = PI - Math.toRadians(edges[i + 1].interiorAngleDeg)
            heading -= exterior
        }
    }
    val closureErrorM = hypot(vertices.last().x - vertices.first().x, vertices.last().y - vertices.first().y)
    return PolygonResult(vertices.dropLast(1), closureErrorM)
}
```

**Валідація перед побудовою**: сума введених внутрішніх кутів n-кутника повинна дорівнювати `(n-2) × 180°`; відхилення понад 1° — попередження ще на етапі форми вводу.

**Після побудови координат**:
- `closureErrorM ≤ 0.02` (2 см) — приймається без попередження (похибка вимірювання рулеткою).
- `0.02 < closureErrorM ≤ 0.10` — жовте попередження + автокорекція нев'язки методом лінійного розподілу похибки по довжині ходу (геодезична компенсація, стандартний прийом при обробці замкненого теодолітного ходу):

```kotlin
fun distributeClosureError(vertices: List<Vertex>, errorVector: Vertex): List<Vertex> {
    val perimeter = vertices.indices.sumOf { i ->
        distance(vertices[i], vertices[(i + 1) % vertices.size])
    }
    var cumulative = 0.0
    return vertices.mapIndexed { i, v ->
        val fraction = cumulative / perimeter
        if (i + 1 < vertices.size) cumulative += distance(vertices[i], vertices[i + 1])
        Vertex(v.x - errorVector.x * fraction, v.y - errorVector.y * fraction)
    }
}
```
- `closureErrorM > 0.10` — контур не приймається, форма блокує збереження з переліком ребер для перевірки.

### 3.2 Площа полігону (формула Гаусса)

```kotlin
fun polygonAreaM2(vertices: List<Vertex>): Double {
    val n = vertices.size
    var area2 = 0.0
    for (i in 0 until n) {
        val j = (i + 1) % n
        area2 += vertices[i].x * vertices[j].y - vertices[j].x * vertices[i].y
    }
    return abs(area2) / 2.0
}
```

fun calculateScreed(floorAreaM2: Double, thicknessMm: Double, norm: ConsumptionNorm): LayerResult {

    val mixMassKg = floorAreaM2 * thicknessMm * (norm.kgPerM2PerMm ?: 0.0)

| `PLASTER_GYPSUM_MANUAL` | 0,85–0,90 | 5 | технічні картки груп Ш2 (ДБН В.2.6-22, дод. В) |
| `PLASTER_GYPSUM_MACHINE` | 0,87 (−3% за рахунок рівномірності машинного нанесення) | 8 | — |
| `PLASTER_CEMENT_SAND_MANUAL` | 1,6–1,7 | 30 | шар тоншим 30мм для цементно-піщаної штукатурки сохне нерівномірно і тріскається — стандартна вимога груп Ш4; понад 50мм — з армувальною сіткою |
| `PLASTER_CEMENT_SAND_MACHINE` | 1,6–1,7 | 30 | — |
| `PLASTER_DECORATIVE_KOROID` | 0,2–0,3 | за зерном | — |

Допустимі відхилення обштукатурених поверхонь від площини — за СНиП 3.04.01 (пряме посилання в ДБН В.2.6-22 п. 6.6): контрольна рейка 2м, допуск залежить від категорії якості поверхні (проста/покращена/високоякісна штукатурка) — редаговане поле в налаштуваннях проєкту, впливає лише на текст рекомендацій, не на витрату матеріалу.

### 4.3 Маяки і профіль

```kotlin
fun beaconsCountForWall(wallLengthM: Double, spacingM: Double = 1.3, cornerOffsetM: Double = 0.2): Int {
    val usable = wallLengthM - 2 * cornerOffsetM
    if (usable <= 0) return 2
    val intermediate = floor(usable / spacingM).toInt()
    return intermediate + 2
}
```

Дефолтний крок 1,3 м — типовий для правила довжиною до 1,5 м (стандартна побутова довжина інструменту); для професійного правила 2 м крок можна виставити до 2 м — редагований параметр, не хардкод.

### 4.4 Ґрунтовка

Обов'язковий проміжний шар — рушій автоматично додає його при переході `SCREED_* → FLOORING_*` і перед будь-яким `PLASTER_*`/`PUTTY_*`/`SCREED_SELF_LEVELING_*`, якщо шар не доданий вручну. Норма — 0,10–0,15 л/м² за прохід, коефіцієнт для пористих основ ×1,5.

### 4.5 Фінішні покриття підлоги

| layerType | коефіцієнт запасу | примітки |
|---|---|---|
| `FLOORING_LAMINATE` | 7% прямий розкрій / 10% діагональ 45° / +3% при неправильній формі приміщення | + підложка (+5% нахлест), + плінтус по периметру |
| `FLOORING_PARQUET` | 8% / 12% | + клей 1,0–1,3 кг/м² |
| `FLOORING_LINOLEUM` | розкрій по ширині рулону (3м/4м), не проста площа | клей контактний 0,3–0,5 кг/м² |
| `FLOORING_TILE` | 10% прямо / 15% діагональ / +5% складна геометрія | клей 4–6 кг/м², затирка за формулою `(a+b)/(a·b) × шов_мм × товщина_мм × 1,6` |
| `FLOORING_SELFLEVELING_FINISH` | 5% | декоративна наливна, 2–5мм |

### 4.6 Стіни — фінішні покриття

- `WALL_PUTTY`: 1,0–1,2 кг/(м²·мм), стандарт 1–2мм, +3% на приміщення з ≥5 кутами.
- `WALL_PAINT`: водоемульсійна 0,10–0,15 л/м² за шар (дефолт 2 шари), фасадна силіконова 0,20–0,30 л/м².
- `WALL_WALLPAPER`: розрахунок рулонів з урахуванням розкладки і запасу на підгонку рисунка (`patternRepeatCm`, параметризовано).
- `WALL_DRYWALL`: листи 1,2×2,5м/1,2×3,0м, профіль CD/UD, крок стояків 0,4/0,6м.

### 4.7 Стеля

`CEILING_STRETCH` (натяжна): полотно = площа стелі + периметр × 0,02 (запас на натяг у багеті). `CEILING_DRYWALL`: аналогічно §4.6, підвіси з кроком 0,6×1,0м.

### 4.8 Гідроізоляція

Обов'язкова перевірка: якщо `room.roomType in [Bathroom, Balcony, KitchenWetZone]` і шар гідроізоляції відсутній — блокуюче попередження при збереженні пирога підлоги (не хардкод-заборона, а явний діалог підтвердження "гідроізоляція відсутня, продовжити?"). `WATERPROOFING_COATING`: 1,3–1,7 кг/м² за 2 шари + підйом на стіни 20–30см.

### 4.9 Утеплення

`INSULATION_LAYER`: обʼєм = площа × товщина, розрахунок за ДБН В.2.6-33:2018 для товщини утеплювача виходить за межі MVP (потребує теплотехнічного розрахунку опору теплопередачі) — у MVP товщина вводиться прорабом вручну за проєктною документацією, застосунок лише рахує кількість матеріалу, не проєктує теплоізоляцію.

## 5. Демонтаж

### 5.1 Знесення стін (`WALL_REMOVAL`)

| матеріал | густина, кг/м³ | виробіток ручний, м²/люд·год | виробіток з відбійним молотком, м²/люд·год |
|---|---|---|---|
| цегла | 1600–1800 | 0,15–0,25 | 0,5–0,8 |
| газобетон/піноблок | 500–700 | 0,4–0,6 | 1,0–1,5 |
| ГКЛ-перегородка | — (площею) | 3–5 | — |
| залізобетон монолітний | 2400–2500 | — | 0,15–0,3 (потрібна алмазна різка арматури) |
| блоки ФБС | 1800–2200 | 0,1–0,2 | 0,3–0,5 |

```kotlin
val volumeM3 = lengthM * heightM * thicknessMm / 1000.0
val debrisMassKg = volumeM3 * densityKgM3 * BULKING_FACTOR   // 1.3–1.5, будсміття "спучується"
val laborHours = (lengthM * heightM) / productivityM2PerHour
```

Для `concrete_monolith` — обов'язкове попередження: демонтаж залізобетонних конструкцій без перевірки на несучу функцію є порушенням ДБН В.1.2-14:2018 щодо конструктивної безпеки і потребує проєкту перепланування — самовільний знос несучої стіни або плити перекриття небезпечний.

### 5.2 Прорізання прорізів (`OPENING_CUT`)

Довжина різу = `2 × (ширина + висота)`. Бетон — алмазне різання, 0,5–1,0 м²/год перерізу; цегла/газобетон — штроборіз+перфоратор, 0,3–0,6 м/год довжини різу. Опційний рядок "підсилення прорізу" (кутник/швелер, довжина `ширина + 0,4м`) — не автоматичний, залежить від того, чи стіна несуча.

### 5.3 Демонтаж штукатурки, плитки, покриттів, стелі

| kind | норма виробітку |
|---|---|
| `PLASTER_REMOVAL` (гіпс) | 4–6 м²/люд·год |
| `PLASTER_REMOVAL` (цементно-піщана) | 1,5–3 м²/люд·год |
| `TILE_REMOVAL` | 2–4 м²/люд·год |
| `SCREED_REMOVAL` | 0,5–1,5 м²/люд·год залежно від товщини |
| `FLOORING_REMOVAL` (ламінат/паркет/лінолеум) | 8–15 м²/люд·год |
| `CEILING_REMOVAL` | ×0,8 від plaster_removal (робота "над головою" повільніша) |

### 5.4 Демонтаж фарби (`PAINT_REMOVAL`)

Найризикованіша за передбачуваністю операція — саме тут ховається сценарій "застряг на фарбі у ванній". Параметри:

```kotlin
data class PaintRemovalParams(
    val areaM2: Double,
    val paintType: PaintType,          // WaterBased | OilBased | EnamelAlkyd | Unknown
    val substrate: Substrate,          // Plaster | Concrete | Tile | Drywall
    val removalMethod: RemovalMethod,  // MechanicalGrinder | HeatGun | ChemicalStripper | Combined

)
```

| тип фарби | метод | виробіток, м²/люд·год |
|---|---|---|
| водоемульсійна | шліфмашина | 3–5 |
| олійна/алкідна емаль (типово старі "мокрі зони") | шліфмашина | 0,5–1,2 |
| олійна/алкідна емаль | будфен | 1,5–2,5 (потребує вентиляції — токсичні випари при нагріві старої фарби) |
| олійна/алкідна емаль | хімічна змивка | 1,0–2,0 + час очікування реакції 20–40хв окремим полем |
| алкідна емаль на кахлі | комбінований | 0,3–0,8 (найважчий випадок — низька адгезія знімача до глазурі) |

Якщо `paintType == Unknown` — рушій бере найгірший (найнижчий) норматив і показує попередження: тип фарби не визначено, розрахунок веде найважчий сценарій; олійна/алкідна фарба зазвичай глянцева, стійка до води з милом і не змивається, на відміну від водоемульсійної.

### 5.5 Агрегація

## 7. Кошторис

### 7.1 Джерела цін

Реальність станом на середину 2026 року: у Епіцентру немає публічного відкритого API для пошуку роздрібних цін для сторонніх застосунків — наявні партнерські інтеграції (KeepinCRM, KeyCRM, Бітрікс24) працюють в інший бік: приймання замовлень із маркетплейсу продавцем, а не пошук цін покупцем. Тому:

1. **Ручне введення ціни** — основний і завжди доступний шлях (`PriceSource.Manual`), пишеться у приватну підколекцію користувача.
2. **Cloud Function-адаптер зовнішніх джерел** — окремий, не критичний для роботи застосунку модуль:
   - парсинг публічної видачі epicentrk.ua за назвою SKU — фонова scheduled Cloud Function, кеш у спільній колекції `priceEntries` (оновлення не частіше разу на добу на SKU, `source = Scraped`). Автоматичний парсинг публічних сторінок без офіційного API — сіра зона щодо умов використання сайту постачальника; реалізується з обмеженою частотою запитів і явним попередженням в адмін-панелі, що це на розсуд користувача, рекомендований режим — ручне оновлення раз на тиждень.
   - альтернатива — партнерські API агрегаторів цін (Price.ua та подібні, доступні через конектори на кшталт ApiX-Drive) — "білий" шлях без парсингу чужого сайту напряму.
3. Кожен `EstimateLine` фіксує `priceSource` і `fetchedAt` — видиме прорабу перед формуванням кошторису для клієнта, щоб не здати застарілу ціну.

### 7.2 Формування кошторису

```
lineTotal = qty * unitPrice
lineTotalWithMarkup = if (applyMarkup && markupPercent > 0) lineTotal * (1 + markupPercent / 100) else lineTotal
grandTotal = Σ lineTotalWithMarkup
```

"15% прорабські" — попередньо заповнене, але завжди редаговане поле `markupPercent`, типово застосовується вибірково: до матеріалів — так, до вже закладеної в розцінку роботи — на розсуд прораба (чекбокс `applyMarkup` на кожному рядку окремо). Публічного відкритого джерела актуальних розцінок на будробіт по регіонах України немає — прораб веде власний довідник розцінок, застосунок його зберігає й пропонує повторно при схожих роботах, але не автозаповнює із зовнішнього джерела.

### 7.3 Експорт кошторису

Генерація PDF — на боці Cloud Function (важка операція, не варто вантажити мобільний CPU і батарею): puppeteer/headless Chrome рендерить HTML-шаблон у PDF, результат кладеться у Cloud Storage, посилання повертається клієнту. XLSX — аналогічно, бібліотека exceljs на Cloud Function. Шаблон — українською, з конфігурованим профілем компанії прораба (назва, контакти, логотип) у налаштуваннях застосунку, брендований елементами дизайн-системи §9 (не хардкод одного бренду).

## 9. Дизайн-система

### 9.1 Концепція

Технологічний глазморфізм (напівпрозорі скляні поверхні, м'які тіні, розмиття фону — `Modifier.blur` / `graphicsLayer` з `RenderEffect` на Android 12+, fallback на напівпрозорий солід-колір з softshadow на старіших версіях) поєднаний з українською національною орнаментикою — не як декоративна накладка "вишиванка на кнопці", а як структурний принцип компонування, узятий безпосередньо з композиційної логіки кролевецького ткацтва.

Кролевецький рушник — тканий візерунок Сумщини, для якого характерні: чіткий геометричний орнамент (ромби, трикутники, зигзаги, меандри, восьмикутні зірки, "дерево життя"), строга ритмічність і симетрія композиції, читання орнаменту горизонтальними ярусними смугами, розділеними тонкими лініями, і система червоних форм на білому тлі (історично давніша традиція — червоний орнамент на білому полотні, пізніша — навпаки). Ці три принципи — **геометрична строгість, ярусний ритм, контраст двох кольорів** — прямо переносяться в UI-мову застосунку, а не використовуються як буквальний орнамент-стікер.

### 9.2 Кольорова палітра

```kotlin
object CubeMasterColors {
    val red = Color(0xFFCE1126)        // "прапорний" червоний, як у кролевецькому орнаменті
    val redDeep = Color(0xFF8B0F1F)    // для акцентів на темному фоні
    val white = Color(0xFFFFFFFF)
    val linen = Color(0xFFF7F3EC)      // теплий білий, імітація небіленого полотна — фон, не чистий #FFFFFF
    val graphite = Color(0xFF1C1C1E)   // основний темний текст/фон темної теми
    val glassSurfaceLight = Color(0x33FFFFFF)  // 20% прозорість для скляних карток на світлому фоні
    val glassSurfaceDark = Color(0x261C1C1E)
    val gold = Color(0xFFC9A227)       // рідкісний акцент — статуси "готово", підсумкові суми кошторису
}
```

Кольори не використовуються як прапор — червоний і білий тут виступають функціонально (акцент/фон), без прямого відтворення державної символіки, що коректно для комерційного продукту.

### 9.3 Орнаментальні мотиви як UI-елементи

- **Розділювачі секцій**: замість суцільної лінії — тонкий геометричний паттерн (повторювані ромби/зигзаг), намальований векторно (Compose `Canvas`/`Path`), товщина лінії 1dp, колір `red` на 40% прозорості — читається як декоративна, але не відволікає.
- **Індикатор прогресу заповнення пирога поверхні**: ярусна смуга — кожен доданий шар підсвічується власним "ярусом" паттерну замість звичайного progress bar, пряма метафора з ярусної композиції рушника.
- **Іконка застосунку і splash screen**: стилізований восьмипроменевий мотив (характерний елемент кролевецького орнаменту) поверх скляної картки з тінню.
- **Картки приміщень/матеріалів**: скляна поверхня (`glassSurfaceLight`/`Dark` + blur фону), тонка рамка-паттерн знизу картки замість звичайної тіні-облямівки.
- Орнамент **ніколи** не використовується у зонах з критичною для сприйняття цифровою інформацією (таблиці розрахунків, суми кошторису) — там чиста типографіка на контрастному фоні, глазморфізм і орнамент — periphery, не центр уваги.

### 9.4 Типографіка

Основний шрифт — з підтримкою повного кириличного набору і виразними цифрами (важливо для таблиць розрахунку): `Manrope` або `Golos Text` (обидва мають повну кириличну підтримку і сучасний геометричний рисунок, що суголосний з геометричністю орнаменту). Табличні дані (суми, кількості) — моноширинні цифри (`tabularNums` feature), щоб колонки цифр вирівнювались.

### 9.5 Темна тема

Обов'язкова, з пріоритетом за замовчуванням для роботи на об'єкті при яскравому сонячному світлі (OLED-контраст читається краще на вулиці, ніж світла тема) — графітовий фон, скляні картки на 15% білій прозорості, червоний акцент трохи приглушений (`redDeep`) для зменшення втоми очей.

## 10. Екрани (Compose Navigation, ключові маршрути)

1. **Проєкти** — список карток проєктів (скляні картки, §9.3).
2. **Приміщення проєкту** — картки кімнат з площею/периметром і статусом заповнення пирога (ярусний індикатор).
3. **Редактор геометрії** — перемикач "прямокутник / довільний контур"; для довільного — таблична форма введення ребер (довжина + кут), синхронізована з інтерактивним превʼю на `Canvas` (контур перемальовується наживо, похибка замикання показується числом і кольором індикатора). Додавання вікон/дверей — дотик по ребру контуру → форма розмірів. Повна підтримка мультитач-жестів (масштаб/панорама) — важливо для роботи на планшеті DOOGEE зі слабким touch-контролером, тестувати окремо на низькочутливих екранах.
4. **Конструктор пирога поверхні** — drag-n-drop список шарів знизу вгору, вибір із пресету/вручну з довідника, інпут товщини з живою підказкою мін./макс. і посиланням на норматив (§1).
5. **Демонтаж** — окрема вкладка кімнати, форма для кожного `kind`, підсумкова картка "обсяг сміття / людино-години / номенклатура".
6. **Зведений розрахунок обʼєкта** — таблиця матеріалів з фільтром по кімнаті/типу робіт.
7. **Кошторис** — таблиця + рядки робіт, поле націнки, кнопка експорту PDF/XLSX (запускає Cloud Function, показує прогрес завантаження результату).
8. **Каталог матеріалів і цін** — редагування норм витрати (з видимим дефолтом і нормативним посиланням поруч), ручних цін, запуск оновлення зовнішніх цін.
9. **Профіль компанії** — реквізити для шаблону кошторису, логотип.

---

## 11. Тестування

- **`cubemaster-core`**: unit-тести геометрії на відомих фігурах (прямокутник, Г-подібна кімната, п'ятикутник із заздалегідь відомою площею через незалежний розрахунок), parity-тест — усі формули §4–§5 покриті таблицею вхід→очікуваний вихід.
- **Room ↔ Firestore синхронізація**: інструментальні тести конфлікт-резолюції (два симульовані пристрої редагують один проєкт офлайн, перевірка коректного merge/попередження).
- **UI**: Compose UI-тести на побудову контуру дотиком, включно з edge-case дуже гострих/тупих кутів.
- **Cloud Functions**: юніт-тести генерації PDF/XLSX на фіксованих вхідних даних (снепшот-тест макета документа).

## 12. Етапи реалізації (для Claude Code)

1. `cubemaster-core` (чистий Kotlin-модуль) — геометричний рушій + формула площі + валідація замикання, юніт-тести на відомих фігурах.
2. Доменні моделі + Room DAO + Firebase проєкт (Auth, Firestore rules, базова структура колекцій §2.1).
3. Довідник матеріалів (§4) як seed-дані з нормативними посиланнями (§1), розрахунковий рушій шарів поверх `cubemaster-core`.
4. Модуль демонтажу (§5).
5. Дизайн-система (§9) — тема Compose, глазморфічні компоненти, орнаментальні елементи як переюзабельні Composable.
6. UI: геометрія (Canvas) → конструктор пирога → зведений розрахунок.
7. Кошторис + Cloud Functions для PDF/XLSX.
8. Адаптер зовнішніх цін (§7.1) — підключається останнім, ручний режим лишається повноцінним фолбеком на кожному попередньому етапі.
9. WorkManager-синхронізація і конфлікт-резолюція, офлайн-first перевірка на реальному пристрої без мережі.
